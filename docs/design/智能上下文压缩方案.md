## 1. 背景与动机

### 1.1 问题

当前 LLM 网关（`gateway.py`）在处理长对话时的行为是：

1. `_estimate_input_tokens()` 估算输入 token 数
2. 如果超过模型 `max_context_length` → **跳过该模型**，尝试下一个候选
3. 如果所有候选模型都超限 → **返回错误**

这意味着：一旦对话历史累积超过上下文窗口，用户会话就断了。客户端（OpenClaw Agent、Claude Code 等）需要自己处理截断，体验不一致。

### 1.2 目标

在网关层实现**透明的智能上下文压缩**：

- 当输入 token 接近模型上下文限制时，自动压缩历史消息
- 对客户端完全透明，不需要修改任何调用方
- 保留最近的消息原文 + 将历史消息压缩为结构化摘要
- 支持增量压缩和缓存复用，避免重复处理
- 多实例部署下共享缓存（Redis）
- **按 API Key 粒度控制开关**
- **压缩后在响应 metadata 中标记**

### 1.3 参考实现

`claude-api/internal/compressor` 已经实现了完整的 Go 版本，核心设计经过验证：

- 9 节 XML 结构化摘要（保留技术细节、代码、用户意图等）
- 增量分块压缩（只压缩新增消息，生成独立摘要块）
- 消息前缀 hash 缓存匹配
- 工具调用配对保护

本方案将其核心思路**移植为 Python 版本**，适配网关的多实例 + Redis 架构。

## 3. 详细设计

### 3.1 配置（config.py）

```python
from dataclasses import dataclass


@dataclass
class CompressConfig:
    """上下文压缩配置"""

    # ---- 全局开关 ----
    enabled: bool = True

    # ---- 触发条件 ----
    # 当估算 token 超过模型 max_context 的此比例时触发压缩
    # 例：模型 200K 上下文，0.75 → 150K 时触发
    token_threshold_ratio: float = 0.75
    # 消息数阈值（与 token 阈值取 OR）
    message_threshold: int = 100

    # ---- 保留策略 ----
    # 保留最近 N 条消息不压缩（原文传给模型）
    keep_message_count: int = 6
    # 工具调用配对保护：最大回溯距离
    max_tool_lookback: int = 10

    # ---- 摘要生成 ----
    # 用于生成摘要的模型（从网关自身的模型池中选）
    summary_model: str = "claude-sonnet-4-5-20250929"
    # 单批摘要最大输入字符数
    max_batch_chars: int = 80000
    # 单个摘要块目标 token 上限
    max_summary_tokens: int = 25000
    # 摘要生成超时（秒）
    summary_timeout: int = 60

    # ---- 缓存 ----
    cache_enabled: bool = True
    # Redis key 前缀
    cache_prefix: str = "hx:llm:compress"
    # 缓存 TTL（秒），默认 24 小时
    cache_ttl: int = 86400

    # ---- 工具结果裁剪 ----
    max_tool_result_length: int = 2000
    keep_head_chars: int = 800
    keep_tail_chars: int = 800
```

### 3.2 API Key 粒度控制

#### 3.2.1 控制机制

利用 `UserApiKey.metadata_` (JSON) 字段存储压缩开关，**不需要加数据库列**：

```json
// UserApiKey.metadata_ 示例
{
  "compress_enabled": true,      // 显式开启
  "compress_threshold_ratio": 0.8 // 可选：覆盖全局阈值
}
```

**三态逻辑**（优先级从高到低）：

| API Key metadata | 全局配置 | 最终行为 |
|---|---|---|
| `"compress_enabled": true` | 任意 | ✅ 开启 |
| `"compress_enabled": false` | 任意 | ❌ 关闭 |
| 无此字段 / metadata 为 null | `LLM_COMPRESS_ENABLED=true` | ✅ 开启（跟随全局） |
| 无此字段 / metadata 为 null | `LLM_COMPRESS_ENABLED=false` | ❌ 关闭（跟随全局） |

#### 3.2.2 判断函数

```python
def is_compress_enabled(
    global_config: CompressConfig,
    api_key_metadata: dict | None,
) -> bool:
    """
    判断是否启用压缩（API Key 级别 > 全局级别）
    """
    if api_key_metadata and "compress_enabled" in api_key_metadata:
        return bool(api_key_metadata["compress_enabled"])
    return global_config.enabled
```

通过现有的 `UpdateUserApiKeyParam` 接口即可管理，无需新增 API：

### 3.3 核心压缩器（compressor.py）

#### 3.3.1 类设计

```python
class ContextCompressor:
    """上下文压缩器"""

    def __init__(self, config: CompressConfig):
        self.config = config
        self.cache = CompressCache(config)  # Redis 缓存

    async def compress_if_needed(
        self,
        messages: list[dict],                # 原始消息列表
        system: str | list | None,           # system prompt
        max_context_length: int,             # 模型上下文长度
        summarizer: Callable,                # 摘要生成回调
        api_key_metadata: dict | None = None, # API Key 级别配置覆盖
    ) -> CompressResult | None:
        """
        如果需要则压缩消息。
        返回 CompressResult（含压缩后消息 + 压缩元信息），
        不需要压缩则返回 None。
        """
```

#### 3.3.2 压缩结果（携带元信息）

```python
@dataclass
class CompressResult:
    """压缩结果"""
    messages: list[dict]          # 压缩后的消息列表
    original_count: int           # 原始消息数
    compressed_count: int         # 压缩后消息数
    summary_block_count: int      # 摘要块数
    original_tokens: int          # 压缩前估算 token
    compressed_tokens: int        # 压缩后估算 token
    cache_hit: bool               # 是否命中缓存
    summary_generation_ms: int    # 摘要生成耗时（ms），缓存命中时为 0
```

#### 3.3.3 压缩判断逻辑

```python
def needs_compression(
    self,
    messages: list[dict],
    system,
    max_context_length: int,
    api_key_metadata: dict | None = None,
) -> tuple[bool, int, int]:
    """
    返回 (是否需要压缩, 估算token数, 消息数)
    """
    token_count = estimate_input_tokens(messages, system)
    message_count = len(messages)

    # API Key 级别可覆盖阈值比例
    ratio = self.config.token_threshold_ratio
    if api_key_metadata and "compress_threshold_ratio" in api_key_metadata:
        ratio = float(api_key_metadata["compress_threshold_ratio"])

    threshold = int(max_context_length * ratio)
    needs = (token_count > threshold) or (message_count > self.config.message_threshold)
    return needs, token_count, message_count
```

#### 3.3.4 消息分割（保护工具调用对）

```python
def split_messages(self, messages: list[dict]) -> tuple[list[dict], list[dict]]:
    """
    分割为 (待压缩历史, 保留的最新消息)
    保证 tool_use 和 tool_result 不被拆开
    """
    if len(messages) <= self.config.keep_message_count:
        return [], messages

    split_idx = len(messages) - self.config.keep_message_count
    split_idx = self._find_safe_split_point(messages, split_idx)

    return messages[:split_idx], messages[split_idx:]
```

`_find_safe_split_point` 的逻辑：
1. 收集保留区域中所有 `tool_result` 的 `tool_use_id`
2. 从分割点向前扫描，找到对应的 `tool_use` 消息
3. 将分割点前移到 `tool_use` 之前，确保配对完整

#### 3.3.5 增量压缩主流程

```python
async def compress_if_needed(self, messages, system, max_context_length,
                              summarizer, api_key_metadata=None):
    # 1. 检查是否需要压缩
    needs, token_count, msg_count = self.needs_compression(
        messages, system, max_context_length, api_key_metadata
    )
    if not needs:
        return None

    # 2. 查 Redis 缓存 —— 尝试复用已有摘要块
    existing_blocks, matched_count = await self.cache.find_matching(messages)
    cache_hit = existing_blocks is not None and matched_count > 0

    # 3. 如果有缓存，先尝试复用
    if cache_hit:
        remaining = messages[matched_count:]
        compressed_msgs = self._build_compressed_messages(existing_blocks, remaining)
        needs2, new_token_count, _ = self.needs_compression(
            compressed_msgs, system, max_context_length, api_key_metadata
        )
        if not needs2:
            return CompressResult(
                messages=compressed_msgs,
                original_count=len(messages),
                compressed_count=len(compressed_msgs),
                summary_block_count=len(existing_blocks),
                original_tokens=token_count,
                compressed_tokens=new_token_count,
                cache_hit=True,
                summary_generation_ms=0,
            )

    # 4. 执行增量压缩
    start_idx = matched_count if cache_hit else 0
    work_messages = messages[start_idx:]
    history, keep = self.split_messages(work_messages)

    if not history:
        if existing_blocks:
            compressed_msgs = self._build_compressed_messages(existing_blocks, work_messages)
            return CompressResult(...)
        return None

    # 5. 生成新摘要（计时）
    t0 = time.monotonic()
    summary = await self._generate_summary(history, summarizer)
    generation_ms = int((time.monotonic() - t0) * 1000)

    # 6. 创建新摘要块
    new_block = SummaryBlock(
        block_id=f"blk_{int(time.time() * 1000)}",
        msg_start_idx=start_idx,
        msg_end_idx=start_idx + len(history),
        summary=summary,
        token_count=estimate_input_tokens([{"role": "user", "content": summary}], None),
        created_at=datetime.now(UTC),
    )

    all_blocks = (existing_blocks or []) + [new_block]

    # 7. 保存到 Redis
    await self.cache.save(messages, start_idx + len(history), all_blocks)

    # 8. 构建压缩后的消息列表
    compressed_msgs = self._build_compressed_messages(all_blocks, keep)
    compressed_token_count = estimate_input_tokens(compressed_msgs, system)

    return CompressResult(
        messages=compressed_msgs,
        original_count=len(messages),
        compressed_count=len(compressed_msgs),
        summary_block_count=len(all_blocks),
        original_tokens=token_count,
        compressed_tokens=compressed_token_count,
        cache_hit=cache_hit,
        summary_generation_ms=generation_ms,
    )
```

#### 3.3.6 压缩后的消息格式

```python
def _build_compressed_messages(
    self, blocks: list[SummaryBlock], keep_messages: list[dict]
) -> list[dict]:
    """构建压缩后的消息列表"""

    summary_parts = [
        "[历史对话摘要 - 结构化压缩]",
        SUMMARY_PREAMBLE,
        "",
    ]
    for i, block in enumerate(blocks):
        summary_parts.append(
            f"=== 摘要块 {i+1} (消息 {block.msg_start_idx+1}-{block.msg_end_idx}) ==="
        )
        summary_parts.append(block.summary)
        summary_parts.append("")
    summary_parts.append("[摘要结束，以下是最近的对话]")

    summary_text = "\n".join(summary_parts)

    compressed = [
        {"role": "user", "content": summary_text},
        {
            "role": "assistant",
            "content": f"好的，我已了解之前的对话上下文（共 {len(blocks)} 个摘要块）。请继续。",
        },
        *keep_messages,
    ]
    return compressed
```

### 3.4 Redis 缓存层（cache.py）

#### 3.4.1 缓存键设计

```
hx:llm:compress:prefix:{prefix_hash}     → JSON (ConversationCache)
hx:llm:compress:idx:{msg_count}           → Set of prefix_hash
```

- **主缓存**：`prefix:{prefix_hash}` 存储完整的 `ConversationCache`（含摘要块列表）
- **消息数量索引**：`idx:{msg_count}` 存储该数量对应的所有 `prefix_hash`，用于快速查找

#### 3.4.2 缓存查找（核心算法）

```python
async def find_matching(
    self, messages: list[dict]
) -> tuple[list[SummaryBlock] | None, int]:
    """
    查找匹配的缓存。从消息数最多的开始匹配（贪心，优先复用最多的摘要）。
    """
    msg_len = len(messages)

    # 获取所有缓存的消息数量（降序排列）
    idx_keys = await redis_client.keys(f"{self.prefix}:idx:*")
    msg_counts = sorted(
        [int(k.rsplit(":", 1)[-1]) for k in idx_keys],
        reverse=True,
    )

    for count in msg_counts:
        if count > msg_len:
            continue

        prefix_hash = self._compute_prefix_hash(messages, count)
        exists = await redis_client.sismember(f"{self.prefix}:idx:{count}", prefix_hash)
        if not exists:
            continue

        cache_data = await redis_client.get(f"{self.prefix}:prefix:{prefix_hash}")
        if not cache_data:
            continue

        cache = ConversationCache.from_json(cache_data)
        return cache.summary_blocks, cache.total_compressed_msg

    return None, 0
```

#### 3.4.3 Hash 计算

```python
def _compute_prefix_hash(self, messages: list[dict], count: int) -> str:
    """计算消息前缀的 SHA256 hash"""
    import hashlib

    parts = []
    for msg in messages[:count]:
        role = msg.get("role", "")
        content = msg.get("content", "")
        if isinstance(content, list):
            content = json.dumps(content, sort_keys=True, ensure_ascii=False)
        elif not isinstance(content, str):
            content = str(content)
        parts.append(f"{role}:{content}")

    raw = "|".join(parts)
    return hashlib.sha256(raw.encode()).hexdigest()
```

#### 3.4.4 缓存保存

```python
async def save(
    self, messages: list[dict], compressed_count: int, blocks: list[SummaryBlock]
):
    prefix_hash = self._compute_prefix_hash(messages, compressed_count)

    cache = ConversationCache(
        prefix_hash=prefix_hash,
        total_compressed_msg=compressed_count,
        summary_blocks=blocks,
    )

    pipe = redis_client.pipeline()
    pipe.set(f"{self.prefix}:prefix:{prefix_hash}", cache.to_json(), ex=self.ttl)
    pipe.sadd(f"{self.prefix}:idx:{compressed_count}", prefix_hash)
    pipe.expire(f"{self.prefix}:idx:{compressed_count}", self.ttl)
    await pipe.execute()
```

### 3.6 摘要生成回调（summarizer）

压缩器不直接调用 LLM，而是接收一个回调函数。在 `gateway.py` 中注入：

```python
# gateway.py 中
async def _create_summarizer(self, db: AsyncSession):
    """创建摘要生成回调，复用网关自身的 LLM 调用能力"""
    async def summarizer(content: str, model: str) -> str:
        import litellm
        response = await litellm.acompletion(
            model=model,
            messages=[{"role": "user", "content": content}],
            max_tokens=25000,
            temperature=0.3,
            timeout=60,
        )
        return response.choices[0].message.content
    return summarizer
```

**为什么不走网关完整链路**：摘要生成是内部调用，不需要 API Key 验证、速率限制、积分扣费（成本平台承担）。直接调 LiteLLM 即可。但需要走网关的供应商密钥解密，所以实际实现中会复用 `_resolve_models` + `_build_litellm_params` 的部分逻辑。

### 4.4 OpenAI 格式的适配

`chat_completion` 和 `chat_completion_stream` 用的是 `ChatCompletionRequest`，消息格式略有不同（有 `system` role）。压缩器统一处理 `list[dict]` 格式，需要一个转换层：

```python
def _openai_messages_to_dicts(self, messages: list[ChatMessage]) -> list[dict]:
    return [msg.model_dump(exclude_none=True) for msg in messages]

def _anthropic_messages_to_dicts(self, messages: list[AnthropicMessage]) -> list[dict]:
    result = []
    for msg in messages:
        d = {"role": msg.role}
        if isinstance(msg.content, str):
            d["content"] = msg.content
        elif isinstance(msg.content, list):
            d["content"] = [
                b if isinstance(b, dict) else b.model_dump(exclude_none=True)
                for b in msg.content
            ]
        result.append(d)
    return result
```

## 7. 边界情况处理

### 7.1 压缩后仍超限

如果压缩后的消息 + 摘要仍然超过上下文限制（极端长对话），处理策略：

1. 尝试**合并多个摘要块为一个更精简的摘要**（二次压缩）
2. 如果仍超限，**减少保留消息数**（从 6 降到 4、2）
3. 最终兜底：保留最后 2 条消息 + 截断摘要

### 7.2 摘要生成失败

摘要调用本身可能失败（超时、模型不可用等）。处理策略：

- **降级**：不压缩，走原有的"跳过超限模型"逻辑
- **日志告警**：记录失败原因，便于排查
- 不阻塞用户请求

```python
try:
    compress_result = await context_compressor.compress_if_needed(...)
except Exception as e:
    log.warning(f'[智能压缩] 压缩失败，降级为原始消息: {e}')
    compress_result = None  # 降级
```

### 7.3 Redis 不可用

缓存是优化项，不是必要项：

- Redis 连接失败 → 跳过缓存，每次重新生成摘要
- 缓存读取异常 → 同上
- 写入失败 → 忽略，下次重新生成

### 7.4 并发压缩

多个实例可能同时对同一对话触发压缩：

- Redis hash 匹配是幂等的（相同输入 → 相同 hash → 相同摘要块）
- 最坏情况：重复生成一次摘要，浪费少量 token，不影响正确性
- **不需要分布式锁**——数据不会不一致，只是可能重复劳动

### 7.5 首条消息就超限

用户的第一条消息就很长（例如贴了大段代码）：

- 此时 `split_messages` 返回 `history=[], keep=全部消息`
- 不触发压缩，走原有逻辑（模型本身可能支持长上下文）

## 8. 成本与性能分析

### 8.1 Token 成本（平台承担）

摘要生成使用 `claude-sonnet-4-5`：

| 场景 | 输入 token | 输出 token | 成本约 |
|------|-----------|-----------|--------|
| 首次压缩 100 条消息 | ~60K | ~5K | ~$0.24 |
| 增量压缩 20 条消息 | ~15K | ~2K | ~$0.06 |
| 缓存命中 | 0 | 0 | $0 |

典型场景：一次长对话压缩 1-2 次，成本 < $0.30，远低于因超上下文导致的对话中断重建成本。

### 8.2 延迟影响

| 操作 | 延迟 |
|------|------|
| 缓存查找（Redis） | ~2ms |
| Hash 计算（CPU） | ~5ms（100 条消息） |
| 摘要生成（LLM 调用） | ~10-30s（首次，依据消息量） |
| 缓存命中复用 | ~10ms |

**首次压缩**会增加 10-30s 延迟，但这发生在对话已经非常长的时候，用户的预期等待时间本来就更长。后续请求缓存命中后几乎无额外延迟。

### 8.3 Redis 内存占用

单个缓存条目：摘要文本 ~5-10KB + 元数据 ~1KB ≈ 10KB。

- 1000 个活跃对话 × 10KB = **10MB**
- TTL 24h 自动过期
- 影响可忽略

## 11. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 摘要丢失关键信息 | 模型回复质量下降 | 9 节结构化摘要 + 保留最近 6 条原文 |

| 压缩引入 bug 影响所有请求 | 服务不可用 | 全局开关 + API Key 级别开关 |
| 平台压缩成本失控 | 财务风险 | 内部调用日志 + Phase 3 成本追踪 |
